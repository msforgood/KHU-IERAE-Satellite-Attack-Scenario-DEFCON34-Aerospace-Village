# 참가자 가이드 (한국어) — 위성의 태양광 패널을 탈취하라

> 확인용 한국어판입니다. 공식/배포본은 영어 `participant-guide.md`이며, 영어판이
> 기준(source of truth)입니다.

환영합니다, 오퍼레이터. 지금 당신은 위성의 명령 업링크에 접근할 수 있습니다.
당신이 보내는 모든 것은 **실제로 통용되는 정상 명령 포맷**입니다 — 단, 값은 당신이 정합니다.

목표 위성은 **DEMOSAT**. 이 위성의 태양광 패널은 태양을 추적하며 전력을 유지합니다.
명령을 보내는 건 버튼 하나로 끝나지 않습니다: **유효한 업링크의 모든 요소**를 올바르게
맞춰야 합니다. 콘솔 왼쪽의 **TARGET INTEL** 도시어를 읽으세요 — 정답이 거기 있습니다.

---

## 업링크 조립 (4개 모두 ✓ 되어야 함)

```
  STEP 1 — TARGET ADDRESSING (타깃 주소 지정)
    도시어의 DEMOSAT와 일치하는 Spacecraft ID(SCID)를 고르세요.
    잘못된 ID → 엉뚱한 위성 → 명령이 무시됩니다.

  STEP 2 — COMMAND SELECT (명령 선택)
    서브시스템과 명령 선택:  [ ADCS ★ ] → adcs_torque
    (위성의 리액션휠 토크)

  STEP 3 — COMMAND VALUE (명령 값)
    토크 슬라이더를 안전 구역(초록)을 지나 빨간 구역까지 — 999 mNm — 민 뒤,
    CONFIRM VALUE를 누르세요.

  STEP 4 — RF CONFIG (RF 설정)
    변조·baud·sample rate를 위성 수신기에 맞추세요(도시어 참고):
    OOK · 100 bps · 24 kSa/s.
    잘못된 RF → 위성이 신호를 복조하지 못합니다.

  → 4개가 모두 잠기면 UPLINK ARMED. [ ⚡ GENERATE UPLINK IQ ]를 누르세요.
    위조한 명령이 무선 신호 파일(attack.cf32)로 변환됩니다.
```

오른쪽 **LIVE CCSDS FRAME**이 진행에 따라 바이트 단위로 조립되는 걸 관찰하세요 —
실제 위성 텔레커맨드가 만들어지는 방식 그대로입니다. SCID를 바꾸면 바이트가 바뀝니다:
엉뚱한 위성을 지정하면 완전히 다른 메시지가 되죠.

---

## 전송하고 관찰하기

```
  attack.cf32를 Virtual Antenna에 로드 → 안테나를 위성에 정렬 → TRANSMIT.
  이제 지상국 화면과 위성을 보라.
  태양광 패널이 태양을 잃고 통제 불능으로 회전한다.
  전력이 붕괴하고, 지상국이 ENERGY SUPPLY CRITICAL을 외친다.
```

## 방금 무슨 일이 일어났나?

이 토크 명령은 **완전히 정상적인 명령**입니다 — 운영자들이 위성을 조준하기 위해
매일 사용하죠. 당신은 어떤 프로토콜도 깨지 않았고, 어떤 보안도 우회하지 않았습니다.
그저 **정상 명령에 악용된 값**을 실었을 뿐입니다.

과도한 토크 → 위성이 자세를 유지하지 못함 → 회전(tumbling) → 태양광 패널이 더 이상
태양을 향하지 못함 → 전력 생성 붕괴 → 배터리 방전.

**이것이 핵심입니다: 정상 명령도 악용되면 공격이 된다.**
그리고 명령이 전달되기까지 얼마나 많은 것이 맞아떨어져야 했는지 주목하세요 — 올바른
위성 주소, 올바른 명령, 올바른 값, 그리고 올바른 RF 파라미터. 명령을 안전하게 보내려면
올바른 바이트만으로는 부족합니다. 그 명령 경로를 이해하는 것이 방어의 첫걸음입니다.

## 더 해볼 것
- 토크를 다시 **초록 안전 구역**으로 되돌려 생성해 보라 — 정상 명령이다. 프레임과
  (경보가 뜨지 않는) 결과를 비교해 보라.
- 일부러 **잘못된 SCID나 RF 설정**을 골라 업링크가 다시 잠기는 걸 보라 — 명령이
  전달될 수 없다.
- 다른 서브시스템(POWER / COMM / OBC)도 살펴보라. 각각 실제 명령이며 고유한 결과를 가진다.

*명령 구조에 대한 설명이 필요하면 부스 운영자에게 문의하세요.*
